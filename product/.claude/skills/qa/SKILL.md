# Q&A — 근거 기반 다중 질문 답변기

사용자가 던진 다중 질문을 근거 기반으로 순차 답변한다.

<HARD-GATE>
rl-verify 검증 단계(Step 3)를 건너뛰지 마라.
비용, 시간, 컨텍스트 길이를 이유로 건너뛰는 것은 절대 허용되지 않는다.
검증 없이 완료를 주장하는 것은 스킬 위반이다.
</HARD-GATE>

## 1. 질문 파싱 & 분류

$ARGUMENTS에서 `##` 헤딩 또는 번호(`1.`, `2.`)로 구분된 질문들을 추출하라.

**mcp__sequential-thinking__sequentialthinking**으로 각 질문을 분류하라:

| 유형 | 패턴 | 근거 전략 |
|------|------|-----------|
| 확인형 | "~하는거지??", "~맞지??" | 로컬 소스에서 확인/반박 → 간결 답변 |
| 동작 원리형 | "어떻게 동작하는거야??" | 소스코드 + 아키텍처 문서 심층 탐색 → 단계별 설명 |
| 정의형 | "뭘 하는거지??", "뭔지" | 로컬 우선 → 부족시 웹 리서치 |
| 비교형 | "차이가 뭐지??", "왜 ~인데??" | 양쪽 근거 수집 → 대비표 |

## 2. 질문별 순차 답변 (미검증 드래프트)

각 질문에 대해 순서대로 답변하라.

**출력 시 반드시 상단에 다음 경고를 포함하라:**
> ⚠️ **미검증 답변** — rl-verify 수렴 검증 전 드래프트입니다. 검증 후 수정사항이 추가됩니다.

### 근거 수집 (Smart Escalation)

아래 순서로 근거를 찾아라. 각 단계에서 충분한 근거를 확보하면 다음 단계로 넘어가지 마라.

1. **로컬 탐색** (항상 먼저): 플러그인/스킬 소스코드, vault 노트, 프로젝트 코드를 Grep, Read, Glob으로 탐색
2. **스킬 에스컬레이션** (필요시):
   - 개념 전체를 깊이 이해해야 할 때 → concept-explainer 스킬 호출
   - 외부 팩트/최신 정보가 필요할 때 → deep-research 스킬 호출
   - 특정 도구/플러그인 질문 → 해당 스킬 호출 (예: ouroboros:help)
3. **웹 리서치** (로컬 부족시): WebSearch + WebFetch

### 답변 포맷

각 질문에 대해 반드시 이 포맷을 따라라:

```
## Q{N}. [질문 제목]

**근거 소스**: [파일 경로:라인번호] 또는 [URL]

[답변 본문]
- 단계별로 설명하라
- 12살도 이해할 수 있게 현실 세계 사물에 비유하라
- 왜 이 답변인지 논리 과정과 판단 기준을 제시하라
- 비교형 질문은 반드시 테이블로 정리하라
- 동작 원리형 질문은 ASCII 다이어그램 또는 흐름도를 포함하라
- 이모티콘을 활용하여 가독성을 높여라 (✅ 장점, ❌ 단점, ⚠️ 주의, 💡 핵심, 📌 참고)

**핵심 정리**: [한 줄 요약]
```

이전 질문의 답변과 연결 관계가 있으면 명시적으로 참조하라.

## 3. rl-verify 수렴 검증 (필수 — 건너뛸 수 없음)

Step 2의 모든 답변을 완성한 후, 반드시 rl-verify를 실행하라.
Tier 판별, 관점 선정, 수렴 기준은 rl-verify 스킬이 자체 결정한다 — 여기서 지정하지 않는다.

### 실행

1. Step 2의 전체 답변 내용을 검증 대상으로 rl-verify를 호출하라
2. rl-verify가 수렴할 때까지 대기하라
3. 검증 결과에서 수정이 필요한 답변이 있으면, Step 2의 답변에 수정 내용을 자연스럽게 녹여서 최종 답변을 다시 출력하라 (diff나 수정 이력 테이블 불필요)

## 내장 규칙

사용자가 별도로 지정하지 않아도 항상 적용하라.

**완료 조건:**
- 모든 질문에 답변할 것
- 각 답변에 근거 소스를 명시할 것
- 질문 간 연결 관계가 있으면 앞 답변을 참조할 것
- **rl-verify 수렴 검증을 실행하고, 수정 필요 시 답변에 녹여서 다시 출력할 것 (필수)**

**하지 말 것:**
- 잘 모르면 사용자에게 질문하라. 임의로 판단하지 마라.
- 한 질문의 답변이 다른 질문과 모순되지 않게 하라.
- 코드를 수정하거나 파일을 생성하지 마라. 읽기 전용으로만 동작하라.
- **rl-verify 검증을 건너뛰지 마라. 비용, 시간, 컨텍스트 길이는 건너뛰는 이유가 되지 않는다.**

**고려 사항:**
- sequential thinking을 활용하라
- Step by step으로 단계별 설명하라
