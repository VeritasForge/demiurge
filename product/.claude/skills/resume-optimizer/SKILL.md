# Resume Optimizer Skill

## Overview

JD(Job Description)와 이력서를 비교 분석하여 **이력서를 실제로 수정**하고 **PDF로 변환**하는 6단계 프로토콜입니다.

```
┌───────────────────────────────────────────────────────────────┐
│              Resume Optimizer Protocol (6-Phase)               │
├───────────────────────────────────────────────────────────────┤
│                                                                │
│  Phase 0: 📋 입력 수집                                        │
│  ┌──────────────────────────────────────────────────┐         │
│  │  이력서 Read + JD WebFetch/파싱 + 출력 경로 확인  │         │
│  └────────────────────┬─────────────────────────────┘         │
│                        │                                       │
│  Phase 1: 🔍 JD vs 이력서 갭 분석                             │
│  ┌────────────────────▼─────────────────────────────┐         │
│  │  필수/우대 매트릭스 + 강점 분석 + AI 톤 초기 분석  │         │
│  │  갭 0개 → Phase 2 건너뛰고 Phase 3 직행            │         │
│  └────────────────────┬─────────────────────────────┘         │
│                        │                                       │
│  Phase 2: 💬 갭 발굴 인터뷰                                   │
│  ┌────────────────────▼─────────────────────────────┐         │
│  │  AskUserQuestion으로 숨은 경험 발굴                │         │
│  │  "이력서 미반영 vs 경험 없음" 구분 완료까지 반복    │         │
│  └────────────────────┬─────────────────────────────┘         │
│                        │                                       │
│  Phase 3: ✍️ 이력서 재작성                                    │
│  ┌────────────────────▼─────────────────────────────┐         │
│  │  팀 리드/IC 목차 선택 + JD 최적화 + 갭 보강       │         │
│  │  → {원본명}-v2.md 생성                             │         │
│  └────────────────────┬─────────────────────────────┘         │
│                        │                                       │
│  Phase 4: 🧹 AI 톤 제거                                      │
│  ┌────────────────────▼─────────────────────────────┐         │
│  │  Skill 도구로 humanize-writing 호출               │         │
│  │  + 한국어 이력서 특화 보충 패스                     │         │
│  └────────────────────┬─────────────────────────────┘         │
│                        │                                       │
│  Phase 5: 📄 PDF 변환                                         │
│  ┌────────────────────▼─────────────────────────────┐         │
│  │  사용자 최종 markdown 확인 → 승인 시 PDF 생성      │         │
│  │  pandoc + weasyprint → {원본명}-v{N}.pdf           │         │
│  └──────────────────────────────────────────────────┘         │
│                                                                │
└───────────────────────────────────────────────────────────────┘
```

**⚠️ 필수 도구**: `mcp__sequential-thinking__sequentialthinking` — 모든 Phase 전환 시 사용
**📖 참조 스킬**: `humanize-writing` — Phase 4에서 Skill 도구로 직접 호출
**📖 참조 스킬**: `job-analysis` — Phase 1 매칭 매트릭스 구조 참조

## Phase 0: 입력 수집

1. `$ARGUMENTS`에서 이력서 경로와 JD를 파싱
2. 이력서를 **Read**로 읽기
3. JD가 URL이면 **WebFetch**로 가져오기, 텍스트면 직접 사용
4. 출력 디렉토리 확인 (이력서 파일과 같은 디렉토리)

**Exit Criteria**: 이력서와 JD가 모두 파싱되고 출력 디렉토리가 확인됨

## Phase 1: JD vs 이력서 갭 분석

**mcp__sequential-thinking__sequentialthinking**을 사용하여 분석 계획을 수립한 후 실행.

### Step 1-1: JD 요구사항 분해

JD를 **주요 업무**, **필수 요건**, **우대 사항** 3개 축으로 분리. 각 항목을 개별 요구사항으로 분해.

> **주의**: "주요 업무"도 반드시 갭 분석 대상에 포함. 주요 업무에 있는 항목(예: A/B 테스트, 데이터 파이프라인)을 누락하면 이력서에서 해당 경험이 빠짐.

### Step 1-2: 매칭 매트릭스 생성

각 JD 요구사항에 대해 이력서에서 매칭되는 경험을 찾아 테이블 출력:

```markdown
### 주요 업무

| 업무 항목 | 매칭 | 이력서 근거 | 보강 필요 사항 |
|---------|------|-----------|--------------|
| [항목] | ✅/🟡/❌ | [해당 경험] | [필요 시] |

### 필수 요건

| 요구사항 | 매칭 | 이력서 근거 | 보강 필요 사항 |
|---------|------|-----------|--------------|
| [항목] | ✅/🟡/❌ | [해당 경험] | [필요 시] |

### 우대 사항

| 요구사항 | 매칭 | 이력서 근거 | 보강 필요 사항 |
|---------|------|-----------|--------------|
| [항목] | ✅/🟡/❌ | [해당 경험] | [필요 시] |
```

매칭 기준:
- ✅ 강함: 직접적이고 구체적인 경험이 이력서에 있음
- 🟡 있지만 약함: 관련 경험은 있지만 깊이나 구체성이 부족
- ❌ 부족: 이력서에서 관련 경험을 찾을 수 없음

### Step 1-3: 강점 분석

JD가 요구하는 것 이상으로 이력서가 강한 부분을 식별. 면접에서 차별화 포인트가 되는 경험을 정리.

### Step 1-4: AI 톤 초기 분석

이력서에서 AI가 작성한 듯한 표현을 미리 식별 (Phase 4에서 상세 교정):
- 괄호 영어 병기 남용
- "~경험을 보유하고 있습니다" 수동태
- 형용사 과잉 ("혁신", "선도", "차세대")
- 키워드 스터핑 (한 문장에 5+ 기술 용어 나열)
- 동일 단어 클러스터링 (같은 단어가 문서에 3+ 반복)

### Step 1-5: 포지셔닝 판단

JD의 직무가 **팀 리드/매니저**인지 **IC(Individual Contributor)**인지 판단. Phase 3의 이력서 목차 선택에 사용.

판단 기준:
- JD에 "팀장", "Lead", "Manager", "팀 리딩" 등 → 팀 리드
- JD에 "개발자", "Engineer", "Developer" (리딩 언급 없음) → IC

**Exit Criteria**: 매칭 매트릭스 완성. ❌/🟡 갭이 1개 이상이면 Phase 2로 진행. 갭이 0개면 Phase 2를 건너뛰고 Phase 3으로 직행.

## Phase 2: 갭 발굴 인터뷰

Phase 1에서 ❌ 또는 🟡로 판정된 갭을 기반으로 **AskUserQuestion**을 사용하여 질문.

### 인터뷰 원칙

- **횟수 제한 없음** — 갭을 충분히 파악할 때까지 유연하게 진행
- 핵심 목표: **"이력서에 안 쓴 것인지 vs 정말 경험이 없는지"** 구분
- 1회 질문에 최대 4개 항목 (AskUserQuestion 도구 제약)
- 사용자 답변에 따라 추가 질문이 필요하면 계속 진행
- 구체적 수치/사례를 최대한 추출 (멘토링 인원, 면접 횟수, 성능 개선 수치 등)
- **인터뷰 마지막에 오픈형 질문 필수**: "이력서에 쓰지 않았지만 이 JD와 관련될 수 있는 경험이 더 있나요?" — AI가 자동으로 식별할 수 없는 "사용자만 아는 경험"을 끌어내는 안전망

### 질문 설계 가이드

각 갭에 대해:
1. **직접 경험 여부** 확인 — 예/아니오가 아닌 구체적 선택지 제공
2. **간접 경험** 가능성 확인 — 관련 업무에서 부분적으로 다뤘는지
3. **구체적 수치/사례** 추출 — "몇 명?", "몇 건?", "어떤 도구로?" 등
4. **회사 귀속 확인** — "이 경험은 어느 회사에서 하셨나요?" 여러 회사 경력이 있으면 귀속 오류 방지 필수

### 인터뷰 결과 정리

```markdown
| 갭 | 실제 경험 | 이력서 반영 여부 | 보강 방향 |
|----|----------|----------------|----------|
| [갭 항목] | ✅ 있음 / ❌ 없음 | ❌ 미반영 / ✅ 반영됨 | [구체적 방향] |
```

**Exit Criteria**: 모든 ❌/🟡 갭에 대해 "이력서 미반영 vs 경험 없음" 구분 완료

## Phase 3: 이력서 재작성

**mcp__sequential-thinking__sequentialthinking**으로 재작성 전략을 수립한 후 실행.

### 출력 규칙

- **항상 새 파일**로 생성. 파일명은 `{원본명}-v{N}.md` (N은 자동 증가). Glob으로 기존 버전 파일을 체크하고, 가장 높은 버전 +1로 생성. v2가 있으면 v3, v3이 있으면 v4.
- 원본 이력서의 **모든 경험이 v2에 포함**되어야 함 (누락 금지)
- Phase 2에서 발굴한 숨은 경험을 반영
- JD 키워드/포지셔닝에 맞춰 최적화
- **프로젝트별 그룹핑(프로젝트명 헤더 + 하위 bullet)을 기본 포맷으로 적용**. 원본에 이미 있으면 존중하여 유지.

### 문장 끝 스타일

- **합니다체 통일**: 모든 bullet의 문장 끝을 "~했습니다", "~줄였습니다", "~없앴습니다" 등 합니다체로 통일
- 명사형 종결("~함", "~없앰", "~줄임")은 사용하지 않음
- 단조로움 방지: "~했습니다"만 반복하지 않고 "~줄였습니다", "~만들었습니다", "~마쳤습니다", "~높였습니다" 등 동사를 다양하게 사용

### 압축 보존 규칙

원본 이력서의 내용을 압축할 때 반드시 지켜야 할 규칙:

1. **별도 시스템/프로젝트는 별도 bullet 유지** — 3개 시스템을 1줄로 뭉개지 않음
2. **정량적 성과(숫자, 비율)는 반드시 보존** — "34% 감소", "189% 향상" 등
3. **의사결정 맥락("A 대신 B를 선택한 이유")은 IC/리드 무관하게 항상 보존** — 기술적 판단력을 보여주는 핵심 정보
4. **JD 키워드와 직접 매칭되는 경험은 축소 금지**

### Diff-First 리뷰 프로세스 (필수)

이력서를 바로 작성하지 않는다. 모든 변경사항을 **3단계 diff**로 먼저 보여주고, 각 단계별 사용자 승인을 받은 후에만 최종 이력서를 작성한다.

#### Diff 1: 구조 변경
현재 목차 → 변경 목차를 Before/After로 비교하여 출력.
- 섹션 추가/제거/이동
- 목차 순서 변경
- 사용자 승인 후 Diff 2로 진행

#### Diff 2: 내용 보강
추가/수정되는 bullet을 **회사별로** Before/After로 비교하여 출력.
- 신규 추가 bullet (Before: 없음 / After: 새 내용)
- 기존 bullet 수정 (Before: 원문 / After: 수정문)
- Scope Line 추가
- 변경 포인트 요약 포함
- 사용자 승인 후 Diff 3로 진행

#### Diff 3: 압축
줄이거나 제거하는 부분을 **회사별로** Before/After로 비교하여 출력.
- 압축 보존 규칙 준수 확인 (별도 시스템=별도 bullet, 정량 성과 보존, 의사결정 맥락 보존)
- 중복 제거 근거 명시
- 압축하지 않는 부분도 명시하여 사용자가 전체 그림을 파악할 수 있도록
- 사용자 승인 후 최종 이력서 작성

> **핵심 원칙**: diff를 통해 사용자가 원본에서 남기고 싶은 부분과 변경되는 구조를 먼저 살펴볼 수 있어야 한다. 임의로 변경해서 결과를 출력하면 개선할 여지를 잃는다.
> **AskUserQuestion reject 시**: 사용자가 "Chat about this"를 선택하면 바로 다음 단계로 넘어가지 말고, 어떤 부분을 논의하고 싶은지 먼저 확인할 것.

### 팀 리드/매니저 포지션용 권장 목차

Phase 1에서 "팀 리드"로 판단된 경우:

```
1. Contact Header (이름, 이메일, GitHub/LinkedIn)

2. Professional Summary (3줄)
   - Line 1: 직무 정체성 + 경력 연수
     - 연차 표기: "N년차" 대신 **"N년+"** 형식 (예: "10년+ 서버 개발 경력")
     - 해당 직무(백엔드/서버) 관련 경력만 카운트. 전체 커리어 깊이는 Experience Summary에서 자연스럽게 드러남
   - Line 2: 핵심 기술 역량 2-3개
   - Line 3: 최대 정량적 성과 1개

3. Core Competencies (리더십 + 기술 혼합, 3-4줄)

4. Experience Summary (경력 타임라인 개요)
   - 회사명 | 직책 | 기간을 역순으로 나열
   - 채용담당자가 10초 안에 경력 흐름을 파악할 수 있도록

5. Work Experience (역순)
   각 역할에 Scope Line 추가:
     "Scope: {팀 규모}명 팀 | {시스템 규모} | {서비스 도메인}"

   2 Pillar로 bullet 구성:
     - People: 채용 프로세스, 팀 빌딩, 멘토링, 성장 지원, 기술 리더십
     - Projects: 프로젝트별로 나열. 비즈니스 성과와 기술 판단이 자연스럽게 섞임.

   > Delivery/Technical 분리는 하지 않음 — 하나의 프로젝트에 성과와 기술 판단이 섞여 있어 분류가 모호해짐.

   Bullet 프레임워크 혼합:
     - XYZ 4개: "성과(X)를 수치(Y)로, 방법(Z)으로 달성" (기술 성과)
     - SOAR 1-2개: "상황→목표→행동→결과" (리더십/전략적 판단)

6. Technical Skills (카테고리별)
   - Languages / Frameworks / Databases / Cloud·Infra / DevOps / AI Tools
   - 10-12개 이내로 제한

7. Education

8. Honors / Certifications
```

### IC 포지션용 권장 목차

Phase 1에서 "IC"로 판단된 경우:

```
1. Contact Header

2. Professional Summary (3줄)
   - Line 1: 연차 표기는 **"N년+"** 형식. 해당 직무 관련 경력만 카운트.

3. Experience Summary (경력 타임라인 개요)

4. Technical Skills (Summary 직후 상단 배치 — ATS 점수 향상)
   - 카테고리별 10-12개 + AI Tools

5. Work Experience (역순)
   2 Pillar로 bullet 구성:
     - People: 코드 리뷰 리딩, 주니어 멘토링, 기술 공유, 온보딩 지원
     - Projects: 프로젝트별로 나열 (프로젝트명 헤더 + 하위 bullet)

   > 시니어 IC도 주니어 성장 지원, 코드 리뷰 리딩, 기술 의사결정 참여 등 People 역량이 중요.

   - XYZ bullet 중심 (역할당 4-6개)
   - 성과 → 수치 → 방법 순서

6. Education

7. Certifications
```

### 한국 기업 특화 조정

- "핵심 역량 3줄 요약" 섹션 추가 가능 (Summary와 Work Experience 사이)
- 사진/나이/주소 제거
- 2-3페이지 허용 (팀 리드/시니어)
- 숙련도 표기 선택적

### 키워드 매핑 패스

이력서 재작성 완료 후, JD의 핵심 키워드(주요 업무/필수/우대에서 추출)가 이력서 본문에 자연스럽게 1회 이상 등장하는지 체크. 키워드가 없으면 관련 경험 bullet에 자연스럽게 삽입.

**Exit Criteria**: 새 파일(`-v{N}.md`, 버전 자동 증가) 생성됨 + 원본의 모든 경험이 포함됨 + 압축 승인 프로세스 완료

## Phase 3.5: 사용자 검토

Phase 3 완료 후, Phase 4(AI 톤 제거)로 진행하기 전에 사용자 확인. **AskUserQuestion**으로:

1. 누락된 경험이 없는지
2. 포맷(프로젝트별 그룹핑, bullet 구조)이 기대에 맞는지
3. 추가하고 싶은 경험이 있는지
4. 압축이 과하거나 부족한 부분이 있는지

사용자 피드백을 반영하여 v2 파일을 수정한 후 Phase 4로 진행.

**Exit Criteria**: 사용자가 v2 이력서의 내용과 구조에 동의

## Phase 3-V: 이력서 검증

Phase 3.5(사용자 검토) 완료 후, Phase 4(AI 톤 제거) **전에** 실행. 검증에서 내용이 수정되면 AI 톤 제거를 다시 할 필요가 없도록.

### Step 3-V-1: 기술적 정확성 검증

**Skill** 도구로 `rl-verify`를 호출:

```
Skill: rl-verify
Args: {이력서 파일 경로} — 이력서에 기재된 기술적 내용의 정확성을 검증해주세요.
검증 관점:
- 기술 용어가 정확하게 사용되었는가
- 정량적 수치가 합리적이고 상호 일관되는가
- 기술 스택 간 조합이 현실적인가
- 아키텍처 설명이 기술적으로 타당한가
```

검증 결과를 **AskUserQuestion**으로 사용자에게 보여주고 3가지 선택지 제공:
1. **"맞습니다, 수정해주세요"** — 이력서 수정
2. **"검증이 틀렸습니다"** — 사용자가 AI 검증의 오류를 잡음 (중요: AI가 잘못 판단할 수 있음)
3. **"그대로 두겠습니다"** — 스킵

수정이 있으면 이력서에 반영 후 Step 3-V-2로 진행.

### Step 3-V-2: 문서 구조/중복/맥락 검증

기술적 정확성이 확정된 후, **Skill** 도구로 `rl-verify`를 다시 호출:

```
Skill: rl-verify
Args: {이력서 파일 경로} — 이력서의 문서 구조, 중복, 맥락 일관성을 검증해주세요.
검증 관점:
- 프로젝트 간 중복된 내용이 있는지
- 시간순/논리적 맥락이 맞는지
- 섹션 간 일관성 (Summary ↔ Work Experience)
- 문장 수준 품질 (불완전한 문장, 어색한 표현, 오타)
```

수정이 있으면 이력서에 반영.

**Exit Criteria**: 기술적 정확성 + 문서 구조 검증 모두 완료, 사용자 승인된 수정 사항 반영됨.

## Phase 4: AI 톤 제거

### Step 4-1: humanize-writing 스킬 호출

**Skill** 도구를 사용하여 `humanize-writing` 스킬을 호출합니다:

```
Skill: humanize-writing
Args: 파일: {v2 이력서 경로} - 이 이력서의 AI 톤을 제거하고 자연스러운 사람의 글로 교정해주세요. 단, 이력서의 프로젝트별 헤더+하위 bullet 구조는 반드시 유지하세요. 프로젝트명 볼드 헤더를 제거하거나 flat bullet으로 변경하지 마세요. 문장 끝은 반드시 합니다체("~했습니다", "~줄였습니다" 등)를 유지하세요. 명사형 종결("~함", "~없앰")로 바꾸지 마세요.
```

> **주의**: humanize-writing의 "볼드+콜론 리스트 단조로움" 규칙은 블로그/에세이용이지, 이력서의 프로젝트별 그룹핑 구조는 AI 톤이 아님. 표준적인 이력서 가독성 패턴.

humanize-writing 실행 후, 변경된 부분의 before/after를 **AskUserQuestion**으로 보여주고 3가지 선택지 제공:
1. **"네, 진행"** — 전체 변경 적용
2. **"일부만 적용"** — 적용할 항목 번호를 입력
3. **"대안 표현 제안"** — 사용자가 직접 대안 표현을 입력

구조적 변경(헤더 제거, bullet 병합 등)은 사용자 승인 없이 적용하지 않음.

humanize-writing 스킬이 8-pass 프로세스를 수행합니다:
1. 구조 (반복 패턴 해소)
2. 과장 (significance inflation 제거)
3. AI 어휘 (Tier 1/2 단어 교체)
4. 문법 (copula avoidance, -ing phrases)
5. 리듬 (문장 길이 변화)
6. 필러/헤징 (불필요한 수식 제거)
7. 연결 (전환어 정리)
8. 개성 (자연스러운 톤 추가)

### Step 4-2: 한국어 이력서 특화 보충 패스

humanize-writing 완료 후, 한국어 이력서에서 추가로 체크할 패턴:

| 패턴 | 설명 | 수정 방향 |
|------|------|----------|
| **괄호 영어 병기 남용** | "고가용성(High-Availability)" 모든 단어에 괄호 | 업계 통용어만 병기. "고가용성 시스템"이면 충분 |
| **수동태 "보유" 패턴** | "~경험을 보유하고 있습니다" | "~했습니다", "~를 만들었습니다" 능동형으로 |
| **형용사 과잉** | "혁신", "선도", "차세대", "극대화" | 구체적 수치나 사실로 대체 |
| **단어 클러스터링** | 같은 단어가 문서에 3+ 반복 ("총괄" 3회, "주도" 5회) | 동의어로 분산하거나 문장 구조 변경 |
| **볼드+콜론 리스트 단조로움** | 모든 bullet이 `**헤더:** 설명` 동일 패턴 | 문단형, 리스트형, 짧은 문장 등 혼합 |
| **문장 끝 단조로움** | 모든 문장이 "~했습니다"로 끝남 | 합니다체 내에서 동사 다양화: "~줄였습니다", "~만들었습니다", "~높였습니다" 등. 명사형("~함", "~없앰")으로 바꾸지 않음 |

> **중요**: "총괄", "주도", "확보" 같은 단어는 **개별적으로는 사람도 많이 씀**. 한 문서에 여러 번 **클러스터링**될 때만 AI 신호. humanize-writing의 Tier 2 판정 기준과 동일.

**Exit Criteria**: humanize-writing 8-pass 완료 + 한국어 보충 패스 완료 + Changes 테이블 출력

## Phase 5: PDF 변환

Phase 4 완료 후, 최종 이력서를 PDF로 변환합니다.

### Step 5-1: 사용자 최종 확인

**AskUserQuestion**으로 사용자에게 최종 markdown 이력서 확인을 요청합니다:

1. 최종 이력서 markdown을 확인하셨나요?
2. PDF로 변환할까요?

사용자가 수정을 요청하면 수정 반영 후 재확인.

### Step 5-2: PDF 생성

**Bash** 도구로 pandoc을 실행하여 PDF를 생성합니다:

```bash
pandoc {v{N}.md 파일 경로} -o {v{N}.pdf 파일 경로} --pdf-engine=weasyprint
```

- 파일명: markdown 파일과 동일한 이름에 `.pdf` 확장자 (예: `jay-heumlabs-v2.md` → `jay-heumlabs-v2.pdf`)
- 출력 디렉토리: 이력서 파일과 같은 디렉토리
- 한글 폰트: weasyprint 기본 설정 사용
- 한글 깨짐 시 fallback: `--pdf-engine=xelatex -V mainfont="AppleGothic"` 옵션으로 재시도

**Exit Criteria**: PDF 파일이 이력서와 같은 디렉토리에 생성됨
