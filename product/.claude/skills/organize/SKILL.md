# 대화 내용 구조화 정리

대화에서 논의한 내용을 **원문을 최대한 보존**하면서 구조화하여 정리한다.

```
Parse the user's input from: $ARGUMENTS
```

## Step 0: 모드 감지 (필수 — 건너뛰지 마라)

`$ARGUMENTS`에서 **파일 경로**를 감지한다.

**감지 규칙** — 다음 중 하나라도 해당하면 파일 경로로 판단한다:
- `@파일명` 형태 (예: `@02-aws-iam-group-concept-explainer.md`)
- `.md`로 끝나는 문자열 (예: `02-aws-iam-group-concept-explainer.md`)
- 슬래시를 포함하는 경로 (예: `wiki/security/02-aws-iam-group.md`)

**판별 순서:**
1. 파일 경로를 감지했는가?
   - No → **생성 모드** (Step C1~C4)로 진행
   - Yes → 2번으로
2. 해당 파일이 존재하는가? (Read 도구로 확인)
   - No → **생성 모드** (Step C1~C4)로 진행
   - Yes → **병합 모드** (Step M1~M6)로 진행. **Step C1~C4는 무시하라.**

⚠️ **병합 모드가 감지되면 생성 모드의 어떤 Step도 참조하지 마라.** 특히 Step C4의 "직접 출력 (파일 저장 없음)"은 병합 모드에 적용되지 않는다.

---

## 생성 모드 (Create Mode)

대화 내용을 새로운 구조화 문서로 만들어 **화면에 출력**한다.

### Step C1: 대상 식별

- `$ARGUMENTS`에 주제 힌트가 있으면: 해당 주제의 대화 내용만 추출
- 주제 힌트가 없으면: 가장 최근에 깊이 논의된 주제 전체를 대상으로 함

### Step C2: 원문 보존 원칙

1. **원문 유지**: 대화에서 사용된 설명, 비유, 코드 예시, 표를 최대한 원문 그대로 사용
2. **중복 병합**: 같은 내용이 반복된 경우 가장 완성도 높은 버전 채택
3. **정정 반영**: 후속 대화에서 정정된 경우 **정정된 버전만** 반영
4. **맥락 보존**: 질문의 맥락이 답변 이해에 필요하면 자연스럽게 녹여넣음

### Step C3: 구조화

- **제목(H1)**: 주제를 명확히 나타내는 제목
- **섹션(H2, H3)**: 논리순으로 배치 (시간순 ❌)
- **배치 순서**: 배경 지식 → 핵심 설명 → 심화 내용 → 정리
- **포맷팅**: 코드 블록(언어 식별자), 마크다운 테이블, 비유/예시 유지

### Step C4: 출력

- 구조화된 마크다운 문서를 **직접 출력** (파일 저장 없음)
- 출력 전후에 부가 설명 없이 문서 내용만 출력

---

## 병합 모드 (Merge Mode)

기존 파일의 **섹션 구조를 유지**하면서, 대화 내용을 각 섹션에 **자연스럽게 녹여넣는다.**

### Step M1: 원문 분석

대상 파일을 **Read 도구**로 읽어서 **섹션 구조**를 파악한다 (H1, H2, H3 계층, frontmatter, Sources 등).

### Step M2: 대화 내용 추출

전체 대화에서 병합할 내용을 스캔하여 추출한다:
- Q&A 답변 내용
- 논의/토론에서 나온 보충 설명
- 정정된 정보 (원문의 부정확한 부분)
- 새로운 근거/출처

**대화 내용과 대상 파일의 관련성 판별:**
- 대화 내용이 대상 파일의 주제와 **직접 관련**있으면 → 해당 내용을 추출하여 병합
- 대화 내용이 대상 파일과 **관련이 적으면** → 사용자에게 알린다: "대화 내용 중 [파일명]과 관련된 항목이 N개입니다. 병합을 진행할까요?"
- 관련 내용이 **전혀 없으면** → 사용자에게 알린다: "현재 대화에서 [파일명]에 병합할 내용을 찾지 못했습니다."

### Step M3: 섹션 매핑

추출된 내용을 원문의 **어떤 섹션에 배치할지** 매핑 테이블을 작성하여 화면에 출력한다.

```
매핑 테이블 예시:
Q1 (Principal 설명)     → 2️⃣ 핵심 개념 — 핵심 규칙 #3 보충
Q3 (Deny 우선)          → 3️⃣ 아키텍처 — 권한 평가 보충
Q6 (Group Principal 에러) → 7️⃣ 주의점 — 흔한 실수 #5 보충
```

### Step M4: 병합 결과 미리보기

병합된 **전체 문서를 화면에 출력**한다. 이 단계에서는 파일을 수정하지 않는다.

**병합 원칙:**

1. **원문 구조 보존**: 기존 섹션 순서, 다이어그램, 표, 코드 블록은 그대로 유지
2. **`> 💡` blockquote로 보충**: 보충 내용은 관련 섹션의 해당 항목 바로 뒤에 `> 💡` 블록으로 삽입
3. **정정 우선**: 원문에 부정확한 부분이 대화에서 정정되었으면 원문 자체를 수정
4. **Sources 추가**: 대화에서 인용한 새 출처를 Sources 섹션에 추가
5. **메타데이터 갱신**: Research Metadata 등 문서 메타 정보도 업데이트

**하지 말 것:**
- 별도 Q&A 섹션을 만들지 않는다 — 반드시 관련 섹션에 녹여넣는다
- 원문의 기존 내용을 삭제하지 않는다 (정정된 경우만 예외)
- 원문의 섹션 순서를 변경하지 않는다

### Step M5: 사용자 확인

미리보기 출력 후 사용자에게 확인을 요청한다:

> "파일에 반영할까요?"

- 사용자가 **승인** → Step M6으로 진행
- 사용자가 **수정 요청** → 요청 사항을 반영하여 Step M4를 다시 수행
- 사용자가 **거부** → 종료 (파일 수정 없음)

### Step M6: 파일 저장 + 결과 요약

**Write 도구로 파일을 저장**하고, 어떤 섹션에 무엇이 추가/수정되었는지 간단히 요약한다.

```
예시:
✅ 파일 저장 완료: wiki/security/02-aws-iam-group-concept-explainer.md
- 2️⃣ 핵심 개념: Principal/Effect 역할 구분, AllUsers 예시 설명 추가
- 3️⃣ 아키텍처: 명시적 Deny 우선순위 상세 설명 추가
- 7️⃣ 주의점: 실수 #5 에러 메시지/Workaround 보충
- 📎 Sources: 공식 문서 3개 추가 (8→10건)
```
